# AGENTS.md

This file provides guidance to coding agents when working with code in this repository.

## Build and Validation Commands

```bash
# Build the entire workspace
cargo build --release

# Build just the geyser plugin (produces a cdylib:
# libyellowstone_grpc_geyser.so on Linux, .dylib on macOS)
cargo build --release -p yellowstone-grpc-geyser

# Check plugin config file validity
cargo run -p yellowstone-grpc-geyser --bin config-check -- --config yellowstone-grpc-geyser/config.json

# Format code
cargo fmt --all

# Lint
cargo clippy --workspace --all-targets

# Run tests for common packages
cargo test -p yellowstone-grpc-proto
cargo test -p yellowstone-grpc-client

# Run benchmarks (requires plugin-bench feature)
cargo bench -p yellowstone-grpc-proto --features plugin-bench
```

## Architecture

This is Yellowstone Dragon's Mouth: a Geyser-based gRPC interface for Solana validators.
It streams real-time blockchain data (slots, blocks, transactions, account updates) via gRPC.

### Crate Structure

- `yellowstone-grpc-geyser`: Solana Geyser plugin
  - `plugin.rs`: Implements `GeyserPlugin` trait; entry point receiving validator events
  - `grpc.rs`: gRPC server implementation, subscription handling, filtering, broadcast fanout
  - `config.rs`: Plugin configuration parsing and validation
  - `metrics.rs`: Prometheus metrics endpoint
  - Includes `config-check` binary for validating config files

- `yellowstone-grpc-proto`: protobuf definitions and generated code
  - `proto/geyser.proto`: gRPC service definition (`Subscribe`, `Ping`, `GetLatestBlockhash`, etc.)
  - `src/plugin/`: plugin-side message types and filtering logic
  - `src/plugin/filter/`: subscription filter implementation (accounts, transactions, blocks)
  - Features include: `convert`, `plugin`, `tonic`, `plugin-bench`

- `yellowstone-grpc-client`: Rust client library for connecting to the gRPC server
  - `GeyserGrpcClient`: Main client with subscribe/health/RPC methods
  - `GeyserGrpcBuilder`: Builder for endpoint, TLS, metadata (`x-token`), and connection options

- `examples/`: client examples in multiple languages (`golang`, `python`, `rust`, `typescript`)
  - `examples/rust` is the Rust example crate in this workspace

### Data Flow

1. Solana validator calls `GeyserPlugin` methods (`update_account`, `notify_transaction`, etc.).
2. Plugin converts events to internal `Message` types and sends them to gRPC service via channels.
3. `GrpcService` applies subscription filters and broadcasts matching updates to connected clients.
4. Clients receive `SubscribeUpdate` messages via streaming gRPC.

### Key Concepts

- Commitment levels: `processed` / `confirmed` / `finalized`.
- Filter limits: configurable in `config.json` to restrict subscription scope.
- Block reconstruction: transactions/accounts are assembled into full block messages.
- `x-token`: optional authentication header for client connections.
- Stream keepalive: clients can reply to server ping messages to keep connections alive through load balancers.

## Transaction-grouped account notifications (`transaction_accounts`)

This fork adds a `transaction_accounts` subscription: all accounts modified by a single
transaction delivered in one grouped `SubscribeUpdateTransactionAccounts` message
(filterable by owner/account, with `include_all_accounts` and `readonly_mints_only` options).

**This feature requires a forked validator.** It is fed by a custom Geyser callback
(`notify_transaction_accounts`) that does not exist in upstream agave. The workspace
`Cargo.toml` `[patch.crates-io]` points the agave/solana crates at the fork
(`github.com/claudicular/agave`, branch `add_tx_with_accounts_geyser`, rebased onto the
matching agave version). When rebasing this repo onto a new release tag, rebase the agave
fork onto the matching agave version first, then update the `[patch]` revision.

**Running it:**

```bash
# Validator must be built from the forked agave and launched with the gating flag
# (off by default for compatibility):
agave-validator \
    --geyser-plugin-config yellowstone-grpc-geyser/config.json \
    --enable-transaction-accounts-notify

# Subscribe (Rust example):
cargo run --bin subscribe-transaction-accounts -- \
    --endpoint http://127.0.0.1:10000 \
    --program 675kPX9MHTjS2zt1qfr1NYHuzeLXfQM9H24wFSUt1Mp8
```

The plugin opts in via `GeyserPlugin::transaction_accounts_notifications_enabled() == true`;
the validator only emits the data when launched with `--enable-transaction-accounts-notify`.
